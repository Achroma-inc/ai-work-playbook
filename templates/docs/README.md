# 知識蓄積の運用ガイド

このプロジェクトでは、作業を通じて得たドメイン知識と設計判断をリポジトリ内に蓄積する。
ファイルを書くのは基本的にAIエージェントで、人間は差分のレビューと承認を担う。

エージェント向けの実際の運用ルールは、リポジトリルートの `CLAUDE.md` に集約されている。
このREADMEは人間向けに「各ファイルが何のためにあるか」「人間として何を見ればよいか」を述べる。

## ファイルの場所

- `docs/knowledge.md` — 案件固有の生きた知識
- `docs/adr/` — 設計意思決定の記録(ADR)。`0000-template.md` は雛形

配置はプロジェクトの構成に合わせて変えてよい。変えた場合の実際のパスは `CLAUDE.md` の
「知識ファイルの場所」に記載され、エージェントはそれを正として参照する。

## ファイルの役割

### knowledge.md — 案件固有の生きた知識

そのプロジェクトでしか通用しないドメイン知識を蓄積する。

- このクライアントの仕様の癖、業務ルール、用語
- APIや外部システムの制限・ハマりどころ
- デプロイや運用の注意点

**常に最新であってほしい知識**を置く。古くなった記述はAIが上書きして最新化する。
履歴は残さない(履歴はGitが持つ)。

### docs/adr/ — 設計意思決定の記録(ADR)

「なぜその設計を選んだか」を、決定ごとに1ファイル・連番で残す。

- **ADRはエージェントが自走で起こす。** 設計上の意思決定があれば、エージェントが
  ドラフトを書いて差分提案する。人は **PR/差分のレビューで内容を確認** し、
  誤りや早すぎる判断があればコミット前に修正させる。
- `0000-template.md` は雛形。最初の決定が出たら `0001-xxx.md` として起こす。
- **コミット済みのADRは書き換えない。** 決定を覆すときは新しいADRを起こし、
  古いものは Status を `Superseded` にする。これで判断の変遷が辿れる。

### knowledge.md と ADR の使い分け

判断基準は **「将来この決定を覆すとき、過去の理由を残す必要があるか?」**。

| | knowledge.md | ADR |
|---|---|---|
| 性質 | 常に最新化する運用知識 | 時系列で積み上げる意思決定 |
| 古くなったら | 上書きする | 書き換えず Superseded にする |
| 例 | 「このAPIは謎の制限がある」 | 「DBにPostgreSQLを選んだ理由と却下案」 |

## 人間がやること

ファイルを書くのはAIだが、**差分のレビューと承認は人間が行う**。

- エージェントが提案した `knowledge.md` / ADR の差分を確認する
- 事実誤認やノイズがあれば修正させてからコミットする

これだけ。書く負担がないので、知識蓄積が形骸化しない。
